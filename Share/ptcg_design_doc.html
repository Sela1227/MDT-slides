Chapter 01

專案定位與範圍

劃清邊界，是工程的第一步。一個說清楚「不做什麼」的專案，比一個野心無邊的清單更可能完工。

使用者與性質

本專案定位為個人作品兼親友圈使用，不涉及商務、不公開營運、不擔心 IP 授權。這個前提解放了一些後續設計選擇——不必處理大規模反作弊、不必處理嚴格的反爬蟲合規、不必過度投資基礎設施。

規則範圍

主動跟隨台灣官方最新發行版本。撰寫此文件時的最新擴充包是 超級進化 M4 · 忍者飛旋（2026 年 3 月發售），屬 Regulation Mark I。

交付優先順序

  1. 資料庫優先：先把卡牌的結構化資料完整建好。沒有資料，引擎無從談起。
  2. 引擎次之：把資料庫的卡片，透過 DSL 編碼成可被引擎執行的規則。
  3. 對戰場景最後：雙人線上即時對戰、配對、客戶端 UI。
設計哲學

這是一個慢工出細活的專案。寧可資料庫做透、DSL 設計清楚，也不要急著弄出個半生不熟的對戰客戶端。前期紮實的資料層會讓後面所有功能都跑得起來；前期偷懶會讓後面寸步難行。

Chapter 02

核心模組總覽

一張對戰平台的解剖圖。每一塊都有自己的複雜度，但只有它們搭在一起才能跑。

系統組成
模組職責
Card Database每張卡的結構化資料：屬性、HP、招式、能量需求、特性、規則文字。
Game Engine回合流程、狀態管理、卡牌效果結算、勝負判定。整個專案最複雜的部分。
Deck Builder組牌與合法性驗證（60 張、同名 4 張上限、Standard 卡池）。
Matchmaking玩家配對、開房、好友對戰、觀戰。
Battle Client UI場面呈現、卡牌互動、動畫、聊天。
Authoritative Server權威伺服器、即時通訊、斷線重連、訊息協議。
Account / Collection帳號、玩家擁有的卡片收藏（如果有的話）。

為什麼引擎是最大的挑戰

寶可夢 TCG 的卡片效果是規則描述自然語言化的。每張卡的招式或特性都可能有獨特的觸發條件、結算邏輯、互動效果。例如：「對方場上每有一隻 ex，這個攻擊就多 60 傷害」、「擲一個硬幣，正面則⋯⋯」、「將你牌堆頂的 5 張⋯⋯」。

這代表你不能寫一個「通用攻擊函式」就解決——每張卡基本上都需要被當成程式碼物件來實作。業界三大常見做法：

  • 腳本化 / DSL：定義一套描述卡片效果的領域特定語言，引擎讀取後執行。新增卡片不用改引擎，但 DSL 設計門檻很高。
  • 程式碼直接實作每張卡：每張卡寫成 class 或 function。彈性最大，但工作量極大。
  • 事件驅動架構：建立統一事件流（造成傷害前、抽牌時、回合開始時），卡片註冊監聽器。Hearthstone、MTG Arena 採用的核心架構。
採用方案

本專案採事件驅動架構 + JSONB DSL的混合策略。引擎核心是事件總線，卡片效果用結構化的 DSL JSON 描述，由 Executor 解讀執行。

Chapter 03

規則版本與環境

標準環境是流動的，規則隨輪替變化。鎖定一個明確的版本作為起點，避免做白工。

2026-27 Standard 環境

2026 年 4 月 10 日（線上版 3 月 26 日）起，G 標記輪替出局。當前合法的 Regulation Mark 是：

當前合法卡池
標記狀態
H合法 — 朱&紫中後期擴充
I合法 — 超級進化系列（含 M4 忍者飛旋）
J合法 — 最新標記
G已輪替出局（2026-04-10）

規則涵蓋面

引擎需要處理的規則層次（疊加）：

  • 朱&紫核心規則：基本能量、ex 卡（擊倒給 2 分）、太晶機制。
  • 超級進化系列規則：Mega ex 卡（被擊倒給 3 分）、Mega 進化當回合不能攻擊。
  • ACE SPEC：牌組僅一張限制的特殊訓練家卡。
  • 狀態異常結算：毒、燒傷、麻痺、睡眠、混亂。互斥性、回合間結算流程。

資料涵蓋順序

  1. 第一階段：H / I / J 三標記的所有 Standard 合法卡。
  2. 第二階段：未來可擴充至全收錄（含舊版退環境的卡）。資料庫 Schema 已預留 is_standard_legal 欄位支援。

Chapter 04

資料庫 Schema

PostgreSQL 主表 + 子表 + 列舉的正規化結構。彈性的部分用 JSONB 承載，需要查詢的維度走關聯欄位。

設計策略

採用主表 + 多型子表 + JSONB 效果欄位的混合結構。原因是 PTCG 卡片效果太自由，純關聯式硬塞會塞不下，但完全 JSON 又會喪失查詢能力。

統一查詢視圖

CREATE VIEW v_cards_full AS
SELECT
  c.*,
  p.hp, p.primary_type, p.stage, p.evolves_from, p.rule_box, p.prize_value,
  p.weakness_type, p.weakness_value, p.resistance_type, p.resistance_value,
  p.retreat_cost, p.is_mega, p.is_tera, p.exception_flags,
  t.subtype AS trainer_subtype,
  COALESCE(t.effect_text_zh, e.effect_text_zh) AS trainer_or_energy_text,
  e.is_basic, e.provides
FROM cards c
LEFT JOIN pokemon_cards p ON c.id = p.card_id
LEFT JOIN trainer_cards t ON c.id = t.card_id
LEFT JOIN energy_cards e ON c.id = e.card_id;
設計重點

effect_status 讓你能追蹤每張卡的 DSL 編碼進度。raw_text_zh_tw 保留爬蟲原文當除錯安全網。exception_flags 是陣列型 JSONB，可用 @> 操作快速查詢。能量需求直接用 JSONB，不拉關聯表，簡潔可查。

Chapter 05

效果 DSL 規格

每張卡的效果都是一段結構化 JSON。引擎讀它，玩家不必看它。設計清楚，後面所有事都好辦。

設計原則

  • 事件驅動：所有效果都歸類為「在某個時機點，做某件事」。最外層是 triggers + effects。
  • 宣告式：DSL 描述「要做什麼」而不描述「怎麼做」，由 Executor 解釋。
  • 原子化操作：所有可能動作拆成最小單元（造成傷害、抽牌、附加能量、改變狀態），組合產生複雜效果。
  • 條件與選擇分離：condition（判斷）與 choose（玩家選擇）走兩套機制。

七大類別

DSL 元素分類
類別用途
Triggers觸發時機（on_attack, on_play, on_evolve, on_turn_start, passive...）
Conditions條件判斷（target_has_tag, energy_attached, coin_flip, prize_count...）
Targets目標選擇（self_active, opponent_bench, choose_target...）
Actions原子動作（damage, heal, draw_card, attach_energy, add_status...）
Modifiers數值修飾（per_count, conditional, fixed_modifier...）
Choices玩家選擇（choose_target, choose_pokemon_in_hand, optional...）
Compositions流程控制（sequence, if_then_else, repeat, optional...）

DSL 演進策略

DSL 不可能一次設計完美，分三階段推進：

  1. 階段一（MVP）：純傷害、簡單條件、硬幣。涵蓋約 60% 的卡。
  2. 階段二：特性、訓練家卡常見效果（抽牌、查找、回收）。涵蓋約 85%。
  3. 階段三：複雜互動（連鎖觸發、被迫選擇、修改其他效果的效果）。長尾。

每階段做完都跑一次「能不能組出當前環境主流牌組」的驗證，確保你做的東西真的有用。

Chapter 06

爬蟲架構

三步分離、失敗可重跑。先打 M4 PoC、再橫掃整個 Standard 卡池。OCR 當備胎。

技術選型

  • 語言：Python 3.11+
  • 動態網頁：Playwright（官方台版站是 SPA）
  • API 呼叫：httpx（如能找到 XHR endpoint 可省下 Playwright 開銷）
  • 資料庫寫入：psycopg[binary]（psycopg3）
  • 資料模型：Pydantic（型別安全的中間 schema）
  • 進度顯示：rich
  • 重試：tenacity

三層分離架構

Step 1 · List Scraper └─ 從 M4 擴充包頁面抓取所有卡片的基礎清單 └─ 輸出: m4_index.json [{ id, name, url, image_url }, ...] Step 2 · Detail Scraper └─ 讀取 m4_index.json，逐張卡訪問詳情頁 └─ 抓取 HP、屬性、招式、效果文字等所有結構化欄位 └─ 輸出: m4_details.json Step 3 · Database Loader └─ 讀取 m4_details.json，轉成 ORM 物件，批次寫入 PostgreSQL └─ effect_code 預設 NULL，等之後人工/半自動補 DSL

三層架構讓你可以單獨重跑某一步。例如資料庫 schema 改了，只需重跑 Step 3。爬蟲解析邏輯改了，只需重跑 Step 2。

解析優先順序

  1. 結構化欄位（HP、屬性、撤退、弱點、抵抗）：HTML 有明確 class，CSS selector 直接撈。
  2. 招式 / 特性：用 DOM 結構切塊，每個招式一個區塊。
  3. 效果文字：純文字保留，存到 raw_text_zh_tw 與 effect_text_zh，不嘗試結構化（DSL 編碼留待後續）。

Fallback：OCR + Vision API

爬不下來的卡（網頁變動、特殊版本）自動加入 ocr_pending.json，後續批次跑 Vision API。Prompt 設計為「請依此 JSON Schema 輸出此卡片資訊」，附上 schema 與卡圖。

資料品質檢查

爬完後跑驗證腳本：

  • 所有寶可夢卡都有 HP？
  • 所有招式都有能量需求？（即使 0 也要記錄）
  • 所有卡片都有 image_url？
  • 同擴充包內 collector_number 沒重複？
  • 寶可夢卡的 evolves_from 是否能在資料庫中找到對應卡？

失敗的卡進入 data_quality_issues.json 待人工審查。

實作順序

  1. 本機裝 PostgreSQL、建立空資料庫、跑 DDL 建表
  2. 實作 Step 1 + 2，跑出 m4_details.json
  3. 人工開幾張卡比對網頁，確認欄位無誤
  4. 實作 Step 3，把 M4 全卡入庫
  5. 手動為 5–10 張 M4 卡編寫 effect_code，驗證 DSL 設計撐得住現實卡片
  6. 穩定後再批次抓 H/I/J 全擴充包

Chapter 07

遊戲狀態模型

GameState 是引擎、伺服器、客戶端三方共識的單一事實來源。設計得乾淨，後面什麼都好辦。

為什麼狀態模型要先設計

GameState 是整場對戰當下的全部資訊的單一資料結構。任何時間點截圖 GameState，理論上就能完整還原當時對局——誰的回合、雙方場面、手牌、牌庫、棄牌堆、能量附加、傷害指示物、剩幾張獎賞。

它的好壞決定三件事：

  • 對引擎：DSL Executor 執行每個 action 時，本質就是讀取 GameState、修改 GameState、產出事件。
  • 對伺服器：權威伺服器要序列化 GameState 給客戶端，但對手隱藏資訊絕對不能傳。GameState 必須能「視角化」。
  • 對客戶端：UI 純粹是 GameState 的渲染層，前端不該有自己的遊戲邏輯狀態。

這個選擇直接影響引擎的可測試性、可重播性。

沒有任何中間紀錄。對局結束你不知道每步發生了什麼，除錯與重播完全做不了。

正解 · 事件流
玩家動作 → DSL Executor
  → 產生 Events
    [DamageDealt, EnergyDiscarded, ...]
  → Reducer 套用到 GameState
  → 可能觸發新 Events（連鎖）
  → 最終 GameState + 事件流

每個 Event 是不可變物件，由獨立 Reducer 套用到 state。可重播、可測試、可除錯。

事件流的好處

  • 自然支援連鎖觸發：Event 被產出時，引擎可檢查「有沒有卡片監聽這個事件」（如「同伴被擊倒時 X」），自動產生新 Events 串起來。
  • 支援 undo / replay：每個 Event 記錄了「reducer 做了什麼變化」，理論上可反向。即使不做 undo，replay 只要從初始 state 依序套用所有 Event 即可。
  • 可測試性高：「給定 GameState + PlayerAction，預期產出哪些 Events」比測試最終 state 直觀得多。
  • 前端動畫天然契合：拿到 Event 流可一個一個播放：能量飛上去 → 傷害數字跳出 → 寶可夢倒下。前後 state 切換則無節奏感。

DSL 執行引擎

把資料庫裡的 effect_code 真正跑起來。引擎是純函式：給 GameState + Action + RNG，產出新 GameState + Event 流。

引擎的職責邊界

釘清楚「該管什麼、不該管什麼」，避免後面職責蔓延。

引擎該管
  • 接收 PlayerAction，驗證合法性
  • 解讀 effect_code 產生 Events
  • 把 Events 套用到 GameState
  • 維護階段流程
  • 處理觸發器與連鎖反應
  • 寫死規則：弱點/抵抗、狀態異常結算、勝負判定
引擎不該管
  • WebSocket 通訊（伺服器層）
  • 視角化序列化（伺服器層）
  • 隨機數源（接受注入的 RNG）
  • 玩家匹配、帳號（外圍系統）
  • UI 動畫（前端）
設計目標

引擎是純函式式的：給定 GameState + PlayerAction + RNG seed，產出新 GameState + Event 流。沒有外部依賴、沒有副作用、易於測試、易於重播。

Effect Executor：DSL 解讀核心

引擎最複雜的部分。每種 action 註冊一個 handler，handler 解讀參數、產生 Events、推到 Event Bus、更新 context。

handle_damage：傷害處理（弱點/抵抗寫死）

def handle_damage(executor, context, action):
    targets = resolve_target(context, action["target"])
    base_amount = action["amount"]
    modifiers = action.get("modifiers", [])

    # 1. 計算修飾後的基礎傷害
    final_amount = base_amount
    for mod in modifiers:
        final_amount = apply_modifier(context, final_amount, mod)

    events = []
    for target in targets:
        damage_after_wr = final_amount

        # 2. 弱點/抵抗計算（寫死規則）
        if not action.get("ignore_weakness"):
            if target.weakness_type == context.source_pokemon.primary_type:
                damage_after_wr *= 2      # 朱&紫：弱點 ×2

        if not action.get("ignore_resistance"):
            if target.resistance_type == context.source_pokemon.primary_type:
                damage_after_wr -= 30     # 朱&紫：抵抗 -30

        # 3. 觸發 BEFORE_DAMAGE，讓監聽器修改傷害
        before_event = BeforeDamageEvent(...)
        modified = executor.event_bus.emit_modifiable(before_event)

        # 4. 產生最終 DamageDealt Event
        events.append(DamageDealtEvent(
            source=context.source_instance,
            target=target.instance_id,
            amount=modified.amount,
            ...
        ))

    for ev in events:
        executor.event_bus.dispatch(ev)
    return events
設計重點

弱點/抵抗在引擎裡寫死，DSL 不需要描述這些。BEFORE_DAMAGE 事件可被監聽器修改——某些卡有「對方造成傷害時減 30」這類效果，本質是監聽 BeforeDamageEvent 並修改 amount。需要兩種事件：可修改的（modifiable）與不可修改的（dispatched）。

def handle_choose(executor, context, action):

本質上是手刻的 coroutine——把同步 DSL 流程變成可中斷可恢復。

Event Bus：事件總線

讓「卡片之間互相影響」變可能的關鍵元件。兩種分派模式：

兩種事件分派
類型用途
Modifiable觸發前可修改：「下次傷害減 30」這類效果。emit_modifiable 回傳監聽器修改後的事件。
Dispatched已成事實的事件：通知監聽器、Reducer 套用，但事件本身不可改。

監聽器註冊：被動效果與特性

寶可夢「特性」常常是被動效果（passive trigger）。當該寶可夢上場時，引擎掃描 abilities，把 passive 監聽器註冊到 Event Bus；離場時自動移除。

經典 bug 來源

「上場時註冊、離場時清理」的生命週期管理很重要——如果忘記清理，已被擊倒的寶可夢的特性還會繼續觸發。這是 TCG 引擎最常見的 bug 之一。

觸發順序的仲裁

當多個監聽器同時觸發時，官方規則是「主動方優先選擇」。但實作上分兩種情境：

自動排序（90%）

監聽器之間沒有實質互動，按固定順序執行結果都一樣。

  1. 主動方的監聽器先
  2. 同方按上場時間順序
  3. 同卡多個監聽器按註冊順序
玩家裁決（10%）

兩個監聽器互相影響、執行順序會改變結果。

  • 都修改同一事件同一欄位
  • 其中一個會讓另一個失效

產生 pending_choice 讓主動方選順序。

State Reducer：套用 Events 到 GameState

設計目標：只有它能改 GameState。其他元件只能透過產生 Event、由 Reducer 套用。

def apply_damage(state, event):
    target_state = state.players[event.target_player].pokemon_states[event.target]
    target_state.damage_counters += event.amount

    # 檢查擊倒：注意這裡不直接處理擊倒，而是產生新事件
    target_card = state.get_card(event.target)
    if target_state.damage_counters >= target_card.hp:
        state.pending_events.append(PokemonKnockedOutEvent(
            target=event.target,
            caused_by=event.event_id,
        ))

注意這個設計：擊倒不在 apply_damage 裡直接處理，而是產生新的 PokemonKnockedOutEvent 排到佇列。這讓「擊倒時觸發特性」（許多卡都有）能透過監聽 PokemonKnockedOutEvent 實現。

def validate_deck(deck_list):
    # 1. 總數 60
    if len(deck_list) != 60:
        return ValidationResult(error="DECK_SIZE_INVALID")

    # 2. 同名卡 ≤ 4 張（基本能量除外）
    name_counts = Counter()
    for card_id in deck_list:
        card = db.get_card(card_id)
        if not card.is_basic_energy:
            name_counts[card.name_zh_tw] += 1
    for name, count in name_counts.items():
        if count > 4:
            return ValidationResult(error=f"TOO_MANY_{name}")

    # 3. ACE SPEC 全牌組僅一張
    ace_specs = [c for c in deck_list if db.get_card(c).is_ace_spec]
    if len(ace_specs) > 1:
        return ValidationResult(error="MULTIPLE_ACE_SPEC")

    # 4. 所有卡都在 Standard 卡池
    illegal = [c for c in deck_list if not db.get_card(c).is_standard_legal]
    if illegal:
        return ValidationResult(error="ILLEGAL_CARDS", details=illegal)

    # 5. 至少一張基礎寶可夢
    has_basic = any(db.get_card(c).stage == "basic" for c in deck_list)
    if not has_basic:
        return ValidationResult(error="NO_BASIC_POKEMON")

    return ValidationResult(ok=True)

Match Session：對局封裝

每場對局是獨立的 asyncio task，封裝引擎、訊息佇列、視角化序列化。

class MatchSession:
    match_id: str
    players: list[str]
    engine: GameEngine
    inbox: asyncio.Queue       # 訊息佇列

    async def run(self):
        while not self.engine.is_game_over():
            try:
                msg = await asyncio.wait_for(self.inbox.get(), timeout=600)
            except asyncio.TimeoutError:
                await self.abort("idle_timeout")
                return

            await self.handle_message(msg)

        await self.finalize()
為什麼用佇列

WebSocket 訊息可能來自兩個 connection（A 和 B），但對局狀態只能被一條執行緒改。佇列確保訊息序列化處理，不會發生兩個玩家同時改 state 的競態。

handle_message 核心流程

async def handle_message(self, msg):
    # 1. 驗證訊息格式
    action = self.parse_player_action(msg)
    if not action:
        await self.send_error(msg.player_id, "INVALID_MESSAGE")
        return

    # 2. 丟給引擎
    try:
        events = self.engine.process_action(self.state, action)
    except IllegalActionError as e:
        await self.send_error(msg.player_id, e.code, e.message)
        return  # 不合法動作不影響 state

    # 3. 取得新 state
    self.state = self.engine.current_state

    # 4. 持久化到 Redis
    await self.persist_state()

    # 5. 視角化後推送給雙方
    for player_id in self.players:
        view = self.state.to_view(player_id)
        view_events = self.filter_events_for(events, player_id)
        await self.send(player_id, {
            "type": "StateUpdate",
            "state": view.serialize(),
            "events": [e.serialize() for e in view_events],
        })

    # 6. 推送給觀戰者（全知視角）
    spectator_view = self.state.to_spectator_view()
    for ws in self.spectators:
        await ws.send_json({...})

Pending Choice 訊息流

1. Player A 宣告攻擊 (PlayerAction: Attack) 2. 伺服器處理，引擎遇到「請選一隻備戰交換」 3. 引擎暫停，pending_choice 寫進 state 4. 伺服器發兩種訊息： → A: StateUpdate (events 到目前為止) + ChoicePrompt → B: StateUpdate (events 到目前為止)，無 ChoicePrompt 5. 為什麼用 Redis 而非 Postgres

對局是「短生命週期、高頻率寫入」的特性，Redis 更適合。純記憶體操作對「每個 action 都寫一次」沒壓力。TTL 自動清理結束的對局。對局結束後想存檔（給重播）再從 Redis 拉到 Postgres。

安全與防作弊（最低標）

  • 所有遊戲邏輯都在伺服器跑：客戶端只送 PlayerAction，所有結算、隨機、判定都在伺服器。
  • 視角化嚴格執行：對手手牌、雙方牌庫順序、未翻獎賞卡絕對不出現在傳給該玩家的訊息裡。
  • 動作合法性伺服器級驗證：客戶端 disable 攻擊按鈕沒用，伺服器收到還是要重新檢查。
  • Token / Session：每連線帶 player_id + token。親友規模用固定 token 即可。
  • 速率限制：每玩家每秒最多 10 個訊息。

關鍵設計：state 與 pendingEvents 分離

伺服器推的 StateUpdate 包含「最新狀態」與「剛剛發生的事件流」。不立即把最新狀態渲染，先把 events 推進 pendingEvents 佇列，由動畫系統依序播放，每播完一個事件才微調渲染中的 state。這讓動畫能「逐步演進」而非「跳到最終結果」。

動畫系統：Event Animation Pipeline

同一個玩家動作可能產生 5–15 個 Events（攻擊 → 觸發特性 → 連鎖傷害 → 擊倒 → 拿獎賞 → 強制換出戰）。一次顯示完玩家會看不懂，所以要排隊播放。

class EventAnimator {
  queue: Event[] = []
  isPlaying = false

  enqueue(events) {
    this.queue.push(...events)
    if (!this.isPlaying) this.playNext()
  }

  async playNext() {
    if (this.queue.length === 0) {
      this.isPlaying = false
      return
    }

    this.isPlaying = true
    const event = this.queue.shift()

    await this.animateEvent(event)

    // inter-event 間隔
    const delay = INTER_EVENT_DELAY[event.type] ?? 200
    await sleep(delay)

    this.playNext()
  }

  async animateEvent(event) {
    switch (event.type) {
      case 'EnergyAttached':
        await this.animateEnergyFly(event.from, event.to)
        store.applyMicroUpdate({ attachEnergy: ... })
        break

      case 'DamageDealt':
        await this.animateAttackFlash(event.source)
        await this.animateDamageNumber(event.target, event.amount)
        store.applyMicroUpdate({ addDamage: ... })
        break

      case 'PokemonKnockedOut':
        await this.animateKnockout(event.target)
        store.applyMicroUpdate({ removePokemon: ... })
        break

      case 'CoinFlipped':
        await this.animateCoinFlip(event.results)
        break
    }
  }
}

Chapter 11

DSL 編寫工作流

1500 張卡的 effect_code 不可能純手寫。LLM 產草稿、自動測試把關、人工驗證——三段式 pipeline 把產能拉到 30–50 卡/小時。

為什麼不能全自動

  • DSL 是新發明的：LLM 沒看過你的 schema，光說明就要塞一大段。即使如此它也會幻想出不存在的 action 名稱。
  • PTCG 規則語言充滿隱性慣例：「將牌庫頂的 5 張⋯⋯」背後有「玩家可看牌但對手不能看」的隱含規則。
  • 錯誤代價高：一張 effect_code 寫錯可能造成「永遠卡住」「無限抽牌」「傷害錯誤」這類嚴重 bug。
採用策略

「LLM 產草稿、人工驗證、自動測試把關」的三段式 pipeline。LLM 做粗活、自動工具做篩選、人工只看真正需要判斷的。

Static Linter：語意檢查

Schema validator 只檢查結構，但抓不出「結構對但邏輯怪」的錯誤：

  • target 是 self_active 但 action 是 damage（要傷害自己嗎？）
  • coin_flip 沒有 on_heads/on_tails（等於什麼都不做）
  • search_deck 後沒有 shuffle_after（違反 PTCG 慣例）
  • 引用了 $chosen 但前面沒有 choose action
def lint_effect_code(code):
    warnings = []

    if has_chosen_reference(code) and not has_choose_action(code):

Docker Compose 整體架構

整個系統打包成 docker-compose 五個服務：

# docker-compose.yml 概念示意
version: '3.9'

services:
  postgres:
    image: postgres:16-alpine
    volumes:
      - postgres_data:/var/lib/postgresql/data
      - ./db/init:/docker-entrypoint-initdb.d:ro
    networks: [internal]
    restart: unless-stopped

  redis:
    image: redis:7-alpine
    command: redis-server --appendonly yes --requirepass ${REDIS_PASSWORD}
    volumes: [redis_data:/data]
    networks: [internal]
    restart: unless-stopped

  backend:
    build: ./backend
    depends_on: [postgres, redis]
    networks: [internal, web]
    restart: unless-stopped

  frontend:
    build: ./frontend  # vite build → nginx serve
    networks: [web]
    restart: unless-stopped

  caddy:
    image: caddy:2-alpine
    ports: ["80:80", "443:443"]
    volumes:
      - ./Caddyfile:/etc/caddy/Caddyfile:ro
      - caddy_data:/data
      - ./card-images:/srv/card-images:ro
    networks: [web]
    restart: unless-stopped

networks:
  internal:  # postgres, redis, backend
  web:       # backend, frontend, caddy

設計重點

  • Caddy 當反向代理而非 nginx。Caddy 自動處理 Let's Encrypt SSL，零設定。
  • 雙網路隔離：internal 給資料庫、web 給對外服務。即使 backend 被入侵，攻擊者也碰不到 postgres/redis。
  • 靜態卡圖獨立 mount：直接給 caddy 服務，不過 backend，省下大量請求負擔。
  • 所有服務 restart: unless-stopped：機器重啟自動拉起所有服務，不用 systemd 額外配置。

Caddyfile 配置

ptcg.example.com {
    handle /static/* {
        root * /srv
        file_server
    }

    handle /card-images/* {
        root * /srv
        file_server
        header Cache-Control "public, max-age=31536000"
    }

    handle /api/* {
        reverse_proxy backend:8000
    }

    handle /ws {
        reverse_proxy backend:8000
    }

        reverse_proxy frontend:80
    }

    log {

            roll_size 100mb

        }
    }
}

資料備份策略

  • Postgres 自動備份：每天凌晨 cron 跑 pg_dump，gzip 壓縮，保留 30 天。
