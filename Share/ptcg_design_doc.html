Chapter 01

專案定位與範圍

劃清邊界，是工程的第一步。一個說清楚「不做什麼」的專案，比一個野心無邊的清單更可能完工。

使用者與性質

本專案定位為個人作品兼親友圈使用，不涉及商務、不公開營運、不擔心 IP 授權。這個前提解放了一些後續設計選擇——不必處理大規模反作弊、不必處理嚴格的反爬蟲合規、不必過度投資基礎設施。

規則範圍

主動跟隨台灣官方最新發行版本。撰寫此文件時的最新擴充包是 超級進化 M4 · 忍者飛旋（2026 年 3 月發售），屬 Regulation Mark I。

交付優先順序

  1. 資料庫優先：先把卡牌的結構化資料完整建好。沒有資料，引擎無從談起。
  2. 引擎次之：把資料庫的卡片，透過 DSL 編碼成可被引擎執行的規則。
  3. 對戰場景最後：雙人線上即時對戰、配對、客戶端 UI。
設計哲學

這是一個慢工出細活的專案。寧可資料庫做透、DSL 設計清楚，也不要急著弄出個半生不熟的對戰客戶端。前期紮實的資料層會讓後面所有功能都跑得起來；前期偷懶會讓後面寸步難行。

Chapter 02

核心模組總覽

一張對戰平台的解剖圖。每一塊都有自己的複雜度，但只有它們搭在一起才能跑。

系統組成
模組職責
Card Database每張卡的結構化資料：屬性、HP、招式、能量需求、特性、規則文字。
Game Engine回合流程、狀態管理、卡牌效果結算、勝負判定。整個專案最複雜的部分。
Deck Builder組牌與合法性驗證（60 張、同名 4 張上限、Standard 卡池）。
Matchmaking玩家配對、開房、好友對戰、觀戰。
Battle Client UI場面呈現、卡牌互動、動畫、聊天。
Authoritative Server權威伺服器、即時通訊、斷線重連、訊息協議。
Account / Collection帳號、玩家擁有的卡片收藏（如果有的話）。

為什麼引擎是最大的挑戰

寶可夢 TCG 的卡片效果是規則描述自然語言化的。每張卡的招式或特性都可能有獨特的觸發條件、結算邏輯、互動效果。例如：「對方場上每有一隻 ex，這個攻擊就多 60 傷害」、「擲一個硬幣，正面則⋯⋯」、「將你牌堆頂的 5 張⋯⋯」。

這代表你不能寫一個「通用攻擊函式」就解決——每張卡基本上都需要被當成程式碼物件來實作。業界三大常見做法：

  • 腳本化 / DSL：定義一套描述卡片效果的領域特定語言，引擎讀取後執行。新增卡片不用改引擎，但 DSL 設計門檻很高。
  • 程式碼直接實作每張卡：每張卡寫成 class 或 function。彈性最大，但工作量極大。
  • 事件驅動架構：建立統一事件流（造成傷害前、抽牌時、回合開始時），卡片註冊監聽器。Hearthstone、MTG Arena 採用的核心架構。
採用方案

本專案採事件驅動架構 + JSONB DSL的混合策略。引擎核心是事件總線，卡片效果用結構化的 DSL JSON 描述，由 Executor 解讀執行。

Chapter 03

規則版本與環境

標準環境是流動的，規則隨輪替變化。鎖定一個明確的版本作為起點，避免做白工。

2026-27 Standard 環境

2026 年 4 月 10 日（線上版 3 月 26 日）起，G 標記輪替出局。當前合法的 Regulation Mark 是：

當前合法卡池
標記狀態
H合法 — 朱&紫中後期擴充
I合法 — 超級進化系列（含 M4 忍者飛旋）
J合法 — 最新標記
G已輪替出局（2026-04-10）

規則涵蓋面

引擎需要處理的規則層次（疊加）：

  • 朱&紫核心規則：基本能量、ex 卡（擊倒給 2 分）、太晶機制。
  • 超級進化系列規則：Mega ex 卡（被擊倒給 3 分）、Mega 進化當回合不能攻擊。
  • ACE SPEC：牌組僅一張限制的特殊訓練家卡。
  • 狀態異常結算：毒、燒傷、麻痺、睡眠、混亂。互斥性、回合間結算流程。

資料涵蓋順序

  1. 第一階段：H / I / J 三標記的所有 Standard 合法卡。
  2. 第二階段：未來可擴充至全收錄（含舊版退環境的卡）。資料庫 Schema 已預留 is_standard_legal 欄位支援。

Chapter 04

資料庫 Schema

PostgreSQL 主表 + 子表 + 列舉的正規化結構。彈性的部分用 JSONB 承載，需要查詢的維度走關聯欄位。

設計策略

採用主表 + 多型子表 + JSONB 效果欄位的混合結構。原因是 PTCG 卡片效果太自由，純關聯式硬塞會塞不下，但完全 JSON 又會喪失查詢能力。

列舉型別

-- 卡片大類
CREATE TYPE card_category AS ENUM ('pokemon', 'trainer', 'energy');

-- 屬性（含無色，共 11 種）
CREATE TYPE pokemon_type AS ENUM (
  'grass', 'fire', 'water', 'lightning', 'psychic',
  'fighting', 'darkness', 'metal', 'fairy', 'dragon', 'colorless'
);

-- 進化階段
CREATE TYPE pokemon_stage AS ENUM (
  'basic', 'stage1', 'stage2', 'vmax', 'vstar',
  'mega', 'baby', 'restored'
);

-- 規則框（決定擊倒給幾分）
CREATE TYPE rule_box_type AS ENUM (
  'none', 'ex', 'mega_ex', 'tera_ex', 'radiant'
);

-- 訓練家子類
CREATE TYPE trainer_subtype AS ENUM (
  'item', 'supporter', 'stadium', 'tool', 'ace_spec'
);

-- 狀態異常
CREATE TYPE status_condition AS ENUM (
  'asleep', 'burned', 'confused', 'paralyzed', 'poisoned'
);

-- Regulation Mark
CREATE TYPE regulation_mark AS ENUM ('F', 'G', 'H', 'I', 'J', 'K');

擴充包與卡片主表

CREATE TABLE expansions (
  id              VARCHAR(10) PRIMARY KEY,            -- 'M4', 'SV10' 等
  name_zh_tw      VARCHAR(100) NOT NULL,
  name_jp         VARCHAR(100),
  name_en         VARCHAR(100),
  release_date    DATE NOT NULL,
  card_count      INTEGER,
  regulation_mark regulation_mark NOT NULL,
  is_standard_legal BOOLEAN NOT NULL DEFAULT TRUE,
  created_at      TIMESTAMPTZ DEFAULT NOW(),
  updated_at      TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE cards (
  id                VARCHAR(20) PRIMARY KEY,           -- 'M4-001'
  expansion_id      VARCHAR(10) NOT NULL REFERENCES expansions(id),
  collector_number  VARCHAR(10) NOT NULL,
  category          card_category NOT NULL,

  name_zh_tw        VARCHAR(100) NOT NULL,
  name_jp           VARCHAR(100),
  name_en           VARCHAR(100),

  rarity            VARCHAR(20),
  illustrator       VARCHAR(100),
  image_url         TEXT,
  image_url_hires   TEXT,

  regulation_mark   regulation_mark NOT NULL,
  is_standard_legal BOOLEAN NOT NULL DEFAULT TRUE,
  reprint_of        VARCHAR(20) REFERENCES cards(id),

  raw_text_zh_tw    TEXT,                            -- 卡面原文（除錯用安全網）

  created_at        TIMESTAMPTZ DEFAULT NOW(),
  updated_at        TIMESTAMPTZ DEFAULT NOW(),

  UNIQUE (expansion_id, collector_number)
);

CREATE INDEX idx_cards_name_zh ON cards(name_zh_tw);
CREATE INDEX idx_cards_category ON cards(category);
CREATE INDEX idx_cards_legal ON cards(is_standard_legal) WHERE is_standard_legal = TRUE;

寶可夢卡子表

CREATE TABLE pokemon_cards (
  card_id           VARCHAR(20) PRIMARY KEY REFERENCES cards(id) ON DELETE CASCADE,

  hp                INTEGER NOT NULL,
  primary_type      pokemon_type NOT NULL,

  stage             pokemon_stage NOT NULL,
  evolves_from      VARCHAR(100),                  -- 從哪隻寶可夢進化（用名字）

  rule_box          rule_box_type NOT NULL DEFAULT 'none',
  prize_value       INTEGER NOT NULL DEFAULT 1,    -- 被擊倒給幾分

  weakness_type     pokemon_type,
  weakness_value    VARCHAR(10),
  resistance_type   pokemon_type,
  resistance_value  VARCHAR(10),

  retreat_cost      INTEGER NOT NULL DEFAULT 0,

  is_mega           BOOLEAN GENERATED ALWAYS AS (rule_box = 'mega_ex') STORED,
  is_tera           BOOLEAN NOT NULL DEFAULT FALSE,

  exception_flags   JSONB DEFAULT '[]'::JSONB,    -- ['immune_to_paralyzed', ...]

  created_at        TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_pokemon_type ON pokemon_cards(primary_type);
CREATE INDEX idx_pokemon_stage ON pokemon_cards(stage);
CREATE INDEX idx_pokemon_rule_box ON pokemon_cards(rule_box);

招式與特性表

CREATE TABLE pokemon_attacks (
  id                BIGSERIAL PRIMARY KEY,
  card_id           VARCHAR(20) NOT NULL REFERENCES cards(id) ON DELETE CASCADE,

  attack_order      INTEGER NOT NULL,
  name_zh_tw        VARCHAR(100) NOT NULL,

  energy_cost       JSONB NOT NULL DEFAULT '{}'::JSONB,    -- {"water": 2, "colorless": 1}

  base_damage       INTEGER,
  damage_modifier   VARCHAR(10),                    -- '+', '×' 或 NULL

  effect_text_zh    TEXT,                            -- 卡面繁中效果
  effect_code       JSONB,                            -- DSL 編碼（核心）

  effect_status     VARCHAR(20) DEFAULT 'pending',    -- 開發進度追蹤

  UNIQUE (card_id, attack_order)
);

CREATE INDEX idx_attacks_card ON pokemon_attacks(card_id);
CREATE INDEX idx_attacks_effect_gin ON pokemon_attacks USING GIN (effect_code);

CREATE TABLE pokemon_abilities (
  id                BIGSERIAL PRIMARY KEY,
  card_id           VARCHAR(20) NOT NULL REFERENCES cards(id) ON DELETE CASCADE,

  name_zh_tw        VARCHAR(100) NOT NULL,
  ability_type      VARCHAR(30) NOT NULL DEFAULT 'ability',

  effect_text_zh    TEXT,
  effect_code       JSONB,
  effect_status     VARCHAR(20) DEFAULT 'pending'
);

CREATE INDEX idx_abilities_card ON pokemon_abilities(card_id);
CREATE INDEX idx_abilities_effect_gin ON pokemon_abilities USING GIN (effect_code);

訓練家與能量子表

CREATE TABLE trainer_cards (
  card_id           VARCHAR(20) PRIMARY KEY REFERENCES cards(id) ON DELETE CASCADE,
  subtype           trainer_subtype NOT NULL,
  effect_text_zh    TEXT NOT NULL,
  effect_code       JSONB,
  effect_status     VARCHAR(20) DEFAULT 'pending'
);

CREATE INDEX idx_trainer_subtype ON trainer_cards(subtype);
CREATE INDEX idx_trainer_effect_gin ON trainer_cards USING GIN (effect_code);

CREATE TABLE energy_cards (
  card_id           VARCHAR(20) PRIMARY KEY REFERENCES cards(id) ON DELETE CASCADE,
  is_basic          BOOLEAN NOT NULL,
  provides          JSONB NOT NULL,
  effect_text_zh    TEXT,
  effect_code       JSONB,
  effect_status     VARCHAR(20) DEFAULT 'pending'
);

統一查詢視圖

CREATE VIEW v_cards_full AS
SELECT
  c.*,
  p.hp, p.primary_type, p.stage, p.evolves_from, p.rule_box, p.prize_value,
  p.weakness_type, p.weakness_value, p.resistance_type, p.resistance_value,
  p.retreat_cost, p.is_mega, p.is_tera, p.exception_flags,
  t.subtype AS trainer_subtype,
  COALESCE(t.effect_text_zh, e.effect_text_zh) AS trainer_or_energy_text,
  e.is_basic, e.provides
FROM cards c
LEFT JOIN pokemon_cards p ON c.id = p.card_id
LEFT JOIN trainer_cards t ON c.id = t.card_id
LEFT JOIN energy_cards e ON c.id = e.card_id;
設計重點

effect_status 讓你能追蹤每張卡的 DSL 編碼進度。raw_text_zh_tw 保留爬蟲原文當除錯安全網。exception_flags 是陣列型 JSONB，可用 @> 操作快速查詢。能量需求直接用 JSONB，不拉關聯表，簡潔可查。

Chapter 05

效果 DSL 規格

每張卡的效果都是一段結構化 JSON。引擎讀它，玩家不必看它。設計清楚，後面所有事都好辦。

設計原則

  • 事件驅動：所有效果都歸類為「在某個時機點，做某件事」。最外層是 triggers + effects。
  • 宣告式：DSL 描述「要做什麼」而不描述「怎麼做」，由 Executor 解釋。
  • 原子化操作：所有可能動作拆成最小單元（造成傷害、抽牌、附加能量、改變狀態），組合產生複雜效果。
  • 條件與選擇分離：condition（判斷）與 choose（玩家選擇）走兩套機制。

七大類別

DSL 元素分類
類別用途
Triggers觸發時機（on_attack, on_play, on_evolve, on_turn_start, passive...）
Conditions條件判斷（target_has_tag, energy_attached, coin_flip, prize_count...）
Targets目標選擇（self_active, opponent_bench, choose_target...）
Actions原子動作（damage, heal, draw_card, attach_energy, add_status...）
Modifiers數值修飾（per_count, conditional, fixed_modifier...）
Choices玩家選擇（choose_target, choose_pokemon_in_hand, optional...）
Compositions流程控制（sequence, if_then_else, repeat, optional...）

三個範例

例 1：基礎傷害招式

{
  "name": "水箭",
  "cost": { "水": 1, "無色": 1 },
  "base_damage": 30,
  "triggers": [{ "type": "on_attack" }],
  "effects": [
    {
      "action": "damage",
      "target": "opponent_active",
      "amount": 30
    }
  ]
}

例 2：條件加傷（對 ex 多 60 傷害）

{
  "name": "獵殺者",
  "cost": { "惡": 2 },
  "base_damage": 80,
  "effects": [
    {
      "action": "damage",
      "target": "opponent_active",
      "amount": 80,
      "modifiers": [
        {
          "type": "conditional",
          "condition": {
            "type": "target_has_tag",
            "target": "opponent_active",
            "tag": "ex"
          },
          "amount": 60
        }
      ]
    }
  ]
}

例 3：複合效果（傷害 + 硬幣 + 狀態）

{
  "name": "電光彈",
  "cost": { "雷": 1 },
  "base_damage": 20,
  "effects": [
    {
      "action": "sequence",
      "steps": [
        { "action": "damage", "target": "opponent_active", "amount": 20 },
        {
          "action": "coin_flip",
          "on_heads": [
            {
              "action": "add_status",
              "target": "opponent_active",
              "status": "paralyzed"
            }
          ]
        }
      ]
    }
  ]
}

DSL 演進策略

DSL 不可能一次設計完美，分三階段推進：

  1. 階段一（MVP）：純傷害、簡單條件、硬幣。涵蓋約 60% 的卡。
  2. 階段二：特性、訓練家卡常見效果（抽牌、查找、回收）。涵蓋約 85%。
  3. 階段三：複雜互動（連鎖觸發、被迫選擇、修改其他效果的效果）。長尾。

每階段做完都跑一次「能不能組出當前環境主流牌組」的驗證，確保你做的東西真的有用。

Chapter 06

爬蟲架構

三步分離、失敗可重跑。先打 M4 PoC、再橫掃整個 Standard 卡池。OCR 當備胎。

技術選型

  • 語言：Python 3.11+
  • 動態網頁：Playwright（官方台版站是 SPA）
  • API 呼叫：httpx（如能找到 XHR endpoint 可省下 Playwright 開銷）
  • 資料庫寫入：psycopg[binary]（psycopg3）
  • 資料模型：Pydantic（型別安全的中間 schema）
  • 進度顯示：rich
  • 重試：tenacity

三層分離架構

Step 1 · List Scraper └─ 從 M4 擴充包頁面抓取所有卡片的基礎清單 └─ 輸出: m4_index.json [{ id, name, url, image_url }, ...] Step 2 · Detail Scraper └─ 讀取 m4_index.json，逐張卡訪問詳情頁 └─ 抓取 HP、屬性、招式、效果文字等所有結構化欄位 └─ 輸出: m4_details.json Step 3 · Database Loader └─ 讀取 m4_details.json，轉成 ORM 物件，批次寫入 PostgreSQL └─ effect_code 預設 NULL，等之後人工/半自動補 DSL

三層架構讓你可以單獨重跑某一步。例如資料庫 schema 改了，只需重跑 Step 3。爬蟲解析邏輯改了，只需重跑 Step 2。

解析優先順序

  1. 結構化欄位（HP、屬性、撤退、弱點、抵抗）：HTML 有明確 class，CSS selector 直接撈。
  2. 招式 / 特性：用 DOM 結構切塊，每個招式一個區塊。
  3. 效果文字：純文字保留，存到 raw_text_zh_tw 與 effect_text_zh，不嘗試結構化（DSL 編碼留待後續）。

Fallback：OCR + Vision API

爬不下來的卡（網頁變動、特殊版本）自動加入 ocr_pending.json，後續批次跑 Vision API。Prompt 設計為「請依此 JSON Schema 輸出此卡片資訊」，附上 schema 與卡圖。

資料品質檢查

爬完後跑驗證腳本：

  • 所有寶可夢卡都有 HP？
  • 所有招式都有能量需求？（即使 0 也要記錄）
  • 所有卡片都有 image_url？
  • 同擴充包內 collector_number 沒重複？
  • 寶可夢卡的 evolves_from 是否能在資料庫中找到對應卡？

失敗的卡進入 data_quality_issues.json 待人工審查。

實作順序

  1. 本機裝 PostgreSQL、建立空資料庫、跑 DDL 建表
  2. 實作 Step 1 + 2，跑出 m4_details.json
  3. 人工開幾張卡比對網頁，確認欄位無誤
  4. 實作 Step 3，把 M4 全卡入庫
  5. 手動為 5–10 張 M4 卡編寫 effect_code，驗證 DSL 設計撐得住現實卡片
  6. 穩定後再批次抓 H/I/J 全擴充包

Chapter 07

遊戲狀態模型

GameState 是引擎、伺服器、客戶端三方共識的單一事實來源。設計得乾淨，後面什麼都好辦。

為什麼狀態模型要先設計

GameState 是整場對戰當下的全部資訊的單一資料結構。任何時間點截圖 GameState，理論上就能完整還原當時對局——誰的回合、雙方場面、手牌、牌庫、棄牌堆、能量附加、傷害指示物、剩幾張獎賞。

它的好壞決定三件事：

  • 對引擎：DSL Executor 執行每個 action 時，本質就是讀取 GameState、修改 GameState、產出事件。
  • 對伺服器：權威伺服器要序列化 GameState 給客戶端，但對手隱藏資訊絕對不能傳。GameState 必須能「視角化」。
  • 對客戶端：UI 純粹是 GameState 的渲染層，前端不該有自己的遊戲邏輯狀態。

頂層結構

GameState ├── meta // 對局元資料 ├── phase // 當前階段 ├── turn // 回合資訊 ├── players[2] // 雙方玩家狀態 ├── stadium // 當前場上的競技場卡（共用） ├── pending_actions // 等待解決的動作堆疊 ├── pending_choices // 等待玩家輸入的選擇 ├── event_log // 歷史事件流（日誌與重播） └── card_instances // 卡片實例註冊表（核心設計）

Card vs Instance：必須分清的兩個概念

這是整個狀態模型最關鍵的設計分野。

Card · 卡片定義

資料庫裡的東西，不可變。一張「皮卡丘 ex」的 HP、招式整場不會變。

所有 PostgreSQL 表都是 Card 層級。

Instance · 卡片實例

這場對戰中、玩家牌組裡的「這張卡的這個個體」。

4 張精靈球 = 4 個 Instance，但都指向同一個 Card 定義。

玩家牌組裡有 4 張皮卡丘 ex：一張在出戰位附 2 能量受 50 傷害；一張在備戰區附 1 能量沒受傷；一張在獎賞卡堆；一張在牌庫深處。4 張卡指向同一個 Card，但每張的當下狀態完全不同——所以需要 InstanceID 區分。

card_instances: {
  "uuid-1": { card_id: "M4-001", owner: "player_a" },
  "uuid-2": { card_id: "M4-001", owner: "player_a" },
  ...
}

# 各區域只存 InstanceID
player_a.deck: ["uuid-3", "uuid-7", ...]
player_a.hand: ["uuid-1", ...]
player_a.active: "uuid-2"

狀態變化：用「事件流」驅動而非「直接改」

這個選擇直接影響引擎的可測試性、可重播性。

反例 · 直接突變
def play_attack(state, ...):
    state.players[d].pokemon_states
        [active].damage_counters += dmg
    if dmg >= hp:
        # 擊倒，移到棄牌堆...
        ...

沒有任何中間紀錄。對局結束你不知道每步發生了什麼，除錯與重播完全做不了。

正解 · 事件流
玩家動作 → DSL Executor
  → 產生 Events
    [DamageDealt, EnergyDiscarded, ...]
  → Reducer 套用到 GameState
  → 可能觸發新 Events（連鎖）
  → 最終 GameState + 事件流

每個 Event 是不可變物件，由獨立 Reducer 套用到 state。可重播、可測試、可除錯。

Event 範例

DamageDealtEvent:
  source_instance: uuid
  target_instance: uuid
  amount: 60
  weakness_applied: true
  resistance_applied: false
  final_amount: 120
  caused_by: "attack:M4-001:thunder_shock"

StatusInflictedEvent:
  target_instance: uuid
  status: "asleep"
  replaced: ["paralyzed"]   # 被互斥規則替換的舊狀態

事件流的好處

  • 自然支援連鎖觸發：Event 被產出時，引擎可檢查「有沒有卡片監聽這個事件」（如「同伴被擊倒時 X」），自動產生新 Events 串起來。
  • 支援 undo / replay：每個 Event 記錄了「reducer 做了什麼變化」，理論上可反向。即使不做 undo，replay 只要從初始 state 依序套用所有 Event 即可。
  • 可測試性高：「給定 GameState + PlayerAction，預期產出哪些 Events」比測試最終 state 直觀得多。
  • 前端動畫天然契合：拿到 Event 流可一個一個播放：能量飛上去 → 傷害數字跳出 → 寶可夢倒下。前後 state 切換則無節奏感。

Pending 機制：玩家輸入的等待狀態

很多效果需要玩家選擇（「選一隻備戰寶可夢交換」「擲硬幣」）。引擎不能繼續往下跑，必須暫停等玩家輸入。

pending_choices: [
  {
    choice_id: "choice-uuid-123",
    awaiting_player: "player_a_id",
    type: "select_pokemon",
    from: ["bench-uuid-1", "bench-uuid-2"],
    count: 1,
    optional: false,
    prompt: "請選擇一隻要交換的備戰寶可夢",
    continuation: "<execution_context_id>"
  }
]

引擎遇到需選擇的 action，產生 pending_choice 寫入狀態、回傳 GameState、暫停。客戶端看到 pending_choices 不為空就跳選擇介面。玩家送回 ChoiceResponse 後引擎從 continuation 接著跑。本質上是把同步 DSL 流程變成可中斷可恢復，類似 coroutine。

權威伺服器持有完整真相 GameState（God's Eye View），但傳給客戶端時必須做視角過濾。

牌庫只見張數只見張數
獎賞卡只見張數（自己也看不到）只見張數

嚴格性提醒

to_view 必須寫得很嚴格——任何漏處理欄位就是潛在作弊點。實務做法是白名單序列化：客戶端不該看到的欄位不存在於 ViewedState 型別定義裡，而非「設成 null」。後者容易因粗心 leak。

序列化策略

用途格式
伺服器內部純 Python dataclass / Pydantic model
Redis 快取orjson（跨語言安全）或 pickle（快）
WebSocket 傳輸JSON，每次傳視角化後的 ViewedState
對局紀錄JSON 存初始 GameState + 完整 Event Stream

狀態大小評估：一場對戰 GameState 約 5–20 KB（JSON），加上 Event Stream 約 20–100 KB。這個量級不需特別優化。

除錯與測試友善設計

  • 每個 Event 帶 caused_by 欄位：指向觸發它的源頭，可追溯整個因果鏈。
  • GameState diff 函式：印「這個 action 導致了哪些欄位變化」比印整個 state 直觀。
  • Builder API 測試輔助：讓測試案例可寫成流暢介面，不用每次手刻完整 GameState。
  • Snapshot 機制：每個玩家動作前 snapshot（開發模式啟用）。對局後段發現問題可二分搜尋。
