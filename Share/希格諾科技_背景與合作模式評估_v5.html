NemoClaw Deep-dive · 核心技術解析

NemoClaw 是什麼？簡報用了它幾分真

希格諾簡報的整個臨床 AI Agent，都建立在 NVIDIA NemoClaw 這套框架上。但 NemoClaw 是 NVIDIA 2026 年 3 月剛發布的開源專案，且目前明確標示為 Alpha / Early Preview。這一節把 NemoClaw 真實面貌完整攤開，並對照希格諾簡報中的能力宣稱。

What it actually is · 它真實的樣子

🦞 NemoClaw 是「OpenClaw 的安全沙箱外殼」

NemoClaw 不是一個臨床 AI 系統，也不是一個成熟的 Agent 平台。它是 NVIDIA 開源的參考工具包（Reference Stack），目的是讓開發者能用一行指令把另一個叫做 OpenClaw 的開源 AI 助理，安全地塞進一個沙箱環境裡跑。 其本身不提供醫療能力、不提供 HIS 介接、不提供合規認證——這些全都要建在它上面、由開發商自己做。

用一個比喻：希格諾簡報展示的是「一棟蓋好的智慧醫院」，但 NemoClaw 提供的只是「鋼筋與安全帽」——非常基礎、非常通用、且還在 alpha 測試中。蓋成醫院還是違章建築，全看希格諾的工程師團隊有多強。

LAYER 4
Agent 框架
NVIDIA NemoClaw

NemoClaw 的工作是「onboard 嚮導 + 安全藍圖 + 狀態管理 + 訊息通道 + 推理路由」——基本上就是一個 CLI 工具加一些 YAML 設定檔，幫你把 OpenClaw 助理裝進沙箱。

它不提供任何醫療專用功能、不處理 HL7 / FHIR、不負責合規。

Built by · NVIDIA · Alpha 階段
LAYER 3
沙箱執行環境
NVIDIA OpenShell

OpenShell 是 NemoClaw 之下的執行環境，也是 NVIDIA 開源專案。提供 Landlock + seccomp + netns 的 Linux 沙箱隔離，控制 Agent 能存取哪些檔案、能對外連到哪些網路。

簡報中所謂的「NVIDIA OpenShell 安全框架 — 所有操作完整記錄」指的就是這層。沙箱本身只能阻止 AI 亂搞，不能保證 AI 做對。

Built by · NVIDIA · 也是新專案
LAYER 2
大語言模型
NVIDIA Nemotron

實際做「理解、生成、判斷」的 LLM。預設用的是 nvidia/nemotron-3-super-120b-a12b，1200 億參數的混合專家模型。

關鍵風險：這個模型沒有醫療專科訓練，是通用模型。要用在臨床判斷需要 fine-tune，希格諾簡報沒有提到他們做了任何醫療領域的微調。

Built by · NVIDIA · 通用模型
LAYER 1 · 底層
運算硬體
NVIDIA DGX Spark / RTX

NemoClaw 官方測試平台是 DGX Spark（一台桌上型 AI 工作站）或 RTX PRO 工作站。希格諾的 SiGTRON AI Factory 預計 2026 下半年才上線。

意思是：在 SiGTRON 上線前，這套系統若要部署到秀傳，得用希格諾自己的小型機房或 Spark 設備。

把希格諾簡報中聲稱的能力，逐條對照 NemoClaw 官方文件的真實能力，可以判斷哪些是合理引用、哪些是過度行銷：

簡報的宣稱 NemoClaw / OpenShell 真實能力 判讀
「NVIDIA OpenShell 安全框架 — 所有操作完整記錄」 OpenShell 確實提供 Landlock + seccomp + netns 沙箱與操作日誌，這部分宣稱大致正確。但要注意：日誌只記錄 Agent 的系統呼叫層級動作（讀檔、發網路請求），不是醫療稽核日誌。要符合醫療法規的稽核需求，希格諾需自己再實作一層。 部分正確
「危險操作 → 自動攔截」（簡報第 5 頁） NemoClaw 提供的是網路與檔案層級的政策控制（例如禁止 Agent 連到某些網域）。但「危險操作」在臨床語境下是「給錯藥、漏掉警示」，這不是 NemoClaw 能判斷的，需要希格諾自己定義業務規則。 誤導性
「Overseer 哨站獨立驗證 AI 的每一筆寫入」（簡報第 13 頁） 「Overseer 哨站」這個名詞在 NVIDIA NemoClaw 文件中找不到。NemoClaw 確實有 SSRF validation 與 blueprint 機制，但稱不上「獨立驗證每筆寫入」這麼強的能力。這很可能是希格諾自己包裝的概念，要在會議上問清楚是誰實作的、原始碼在哪。 術語不實
「NemoClaw・Nemotron・NeMo Agent」三套整合（簡報第 5 頁） 這是 NVIDIA Agent Toolkit 的標準組合，引用無誤。但引用 NVIDIA 套件不等於有能力做應用——就像引用「我們用 Word」不代表能寫出好論文。 合理引用
「混合安全部署 — 院內設備 + Sigtron 私有雲」 NemoClaw 設計上支援本地與雲端推理，可以路由到本地 Nemotron 模型或雲端 API，這部分技術可行。但 Sigtron 私有雲在 2026 下半年才上線，目前談「混合部署」是規劃而非現實。 時程未到
「能力一至六」全套臨床功能（病患查詢、交班摘要、語音記錄、量表填寫、通知歸納、醫師動態） NemoClaw 完全沒有提供任何臨床功能。這六大能力100% 必須由希格諾自己開發，包括介接 HIS、定義業務邏輯、訓練醫療領域模型。簡報的圖示是 AI 生成的概念圖，不是產品截圖。 概念非實品
使用 NVIDIA 商標 / Inception logo NVIDIA Inception 是免費新創計畫，不代表 NVIDIA 對產品的技術背書。NemoClaw 是 Apache 2.0 開源專案，任何人都能用。簡報多次出現 NVIDIA logo 容易讓人誤以為是 NVIDIA 投資或推薦。 商標誤導
⚠ NemoClaw 章節結論

NemoClaw 是工具，不是產品；簡報把工具當產品賣

把 NemoClaw 看清楚之後，希格諾簡報的真實樣貌是：「我們會用 NVIDIA 在 1.5 個月前剛發布的 alpha 版工具包，幫你做一個臨床 AI Agent」。這在技術上是可行路線，但跟簡報營造的「成熟系統」印象有相當大的落差。

對醫院的實際意義：(1) 希格諾必須自己寫的程式碼遠多於簡報暗示的份量；(2) NemoClaw 隨時可能改 API（NVIDIA 官方明說），所以希格諾需要持續維護才能跟上；(3) 簡報中的「Overseer 哨站」等專有名詞需要希格諾說明是不是他們自己加的。

會議追問重點：請對方展示一個現有的 NemoClaw demo（不是簡報，而是真的跑起來的東西），即使只是「龍蝦助理」最簡版本也好。能跑出來代表他們確實掌握這套工具，跑不出來就只是 PPT 公司。

Question

Why this matters

新技術的關鍵不是先進，而是有沒有同伴。如果秀傳是孤島，出事時無人聲援、平時沒有同儕學習對象。如果有同伴，至少還有產業共識。這題的答案會直接決定合作該不該繼續。

五種合作架構（含龍蝦助理 PoC）

考慮到希格諾本業為毫米波雷達 + 算力整合，醫療為新跨界的雙軌特性，搭配 James 主動提議的「龍蝦助理 PoC」場景，建議從低風險、可逆、研究導向的合作切入。

A · 最低風險

非接觸式生命徵象研究合作

Radar-based Vital Signs · 回到他們真正的本業

跳過爭議大的「LINE × HIS 整合」，回頭利用希格諾原本的雷達技術。在病房或長照單位部署毫米波雷達，做非接觸式呼吸、心跳、跌倒偵測——這是他們真正有 SDK、有產品的領域。

  • 選一間病房或長照單位試裝 5-10 套
  • 對照穿戴裝置 / 護理觀察值，做準確度驗證
  • 產出共同研究論文、申請科技部計畫
  • 不接 HIS、不寫病歷、不碰個資
合規難度低
技術成熟度高（本業）
投入成本低
時程3-6 個月
B · 研究合作

術後恢復遠端監測研究

Post-op Monitoring × IRB Study · 對接秀傳手術強項

秀傳體系本身手術量大、IRCAD 訓練中心國際知名。將希格諾雷達裝在術後病房床頭，連續監測呼吸頻率、體動、起身次數，觀察是否能早期偵測併發症（譫妄、肺部併發症、跌倒風險）。

  • 由秀傳 IRB 主導、希格諾提供硬體
  • 聚焦於「資料採集 + 演算法驗證」階段
  • 不寫入病歷、僅作研究資料庫
  • 論文成果若佳 → 為日後正式產品鋪路
合規難度中（IRB）
技術成熟度中
投入成本中
時程6-12 個月
C · 龍蝦助理

單一指標 AI Agent PoC

Lobster Assistant · James 主動提議的場景具體化

James 提的「龍蝦助理」要落地，需先選定一個明確指標。建議候選：

  • 術後 24h 早期警示分數（NEWS2 / MEWS 計算）：自動讀取 vital signs 計算分數、產出建議
  • 跌倒風險評估彙整：整合 Morse Scale 各欄位，AI 產出草稿給護理師確認
  • 輸液速率與藥物交互作用提醒：純查詢、不寫入

三個都是「讀取 + 計算 + 建議」，AI 不直接寫入病歷，護理師在 HIS 內手動確認後才生效。

合規難度中
技術成熟度低（preview）
投入成本中
時程6-9 個月
D · 行政應用

內部行政 AI 助手

Admin Agent · 不碰病歷、只碰行政流程

把簡報中的「IM + AI Agent」概念剝離臨床場域，先在非醫療區（總務、人資、設備管理）試行 NemoClaw 框架——例如「會議室預約助手」、「設備故障報修整理」、「教育訓練紀錄彙整」。

  • 使用院內 IM（Teams / Mattermost）取代 LINE
  • 避開個資法、醫療法的紅線
  • 驗證希格諾 Agent 框架的真實能力
  • 若效果好，再評估擴大到臨床支援
合規難度低
技術成熟度低
投入成本中
時程3-6 個月
E · 高風險長線

單一病房臨床 AI Agent 試辦

Single-Ward Clinical Pilot · 簡報原案的縮小版

如果未來要走簡報原案（IM + AI 整理護理紀錄），必須限縮場域與寫入權限：選一個病房（10-20 床），AI 只負責「讀取彙整、產出草稿」，絕不直接寫入正式病歷，所有寫入需護理師於 HIS 內手動確認。

  • 強制要求：合規白皮書、IRB、資安稽核三件套
  • 用院內私有 IM 取代 LINE
  • 3 個月評估，KPI 不達標立即終止
  • 事故責任、原始碼託管、退場條款全寫進合約

總結：希格諾跟 HIMSS 國際大廠 不在同一個位階

希格諾科技是 2024 年才成立的高雄新創，真正的本業是毫米波雷達訊號處理與 AI 模型，2025 年 Q4 透過 30 億的 SiGTRON AI-FACTORY 計畫一舉成為媒體焦點。 但同時間的 HIMSS 2026 上，Epic 已有 85% 客戶用 AI、Microsoft Dragon Copilot 已有 10 萬名臨床醫師每日使用、Abridge 已部署 150+ 美國醫療系統。 這些成熟廠商有 BAA、有 SOC 2、有具名落地醫院、有量化成效。希格諾在所有這些維度上都尚未啟動，卻提了一份規模相當的提案給秀傳。

真實能力
雷達 + 算力

本業可信，作為感測 / 算力夥伴沒問題。當醫療資訊系統廠商會踩雷。

建議起手式
A + 龍蝦 C

方案 A（雷達生命徵象）+ 方案 C（龍蝦助理單一指標）並行 PoC。

核心建議：把希格諾當作「雷達感測 + 算力夥伴」而非「護理資訊系統供應商」來談合作。如果秀傳真的需要 ambient nursing AI，直接找台灣現有 HIS 廠商或 Microsoft Azure / Abridge 等成熟方案是更務實的選擇。希格諾的價值在他們的雷達本業，不在追逐 NemoClaw 這個還沒長大的賽道。
